Add a new Admin module under MANAGEMENT called: Data Exchange

Important rules:
- Do not redesign the whole admin app
- Do not break existing Sponsors, Attendees, Meetings, Reports, webhook, or CSV import/export workflows
- Preserve all current working functionality
- This is a new admin utility module for importing/exporting data in CSV format

Goal:
Create a centralized Data Exchange section for:
- Sponsors
- Attendees
- Meetings

This section must support:
- CSV export
- CSV import
- downloadable sample CSV templates
- filters on export
- validation on import
- duplicate prevention
- row-level rejection reporting

==================================================

Each category must support CSV upload/import.

Flow:
1. Upload CSV
2. Parse file
3. Validate headers
4. Preview import summary
5. Show valid rows / invalid rows
6. Confirm import
7. Show final results

Do NOT directly import without validation.

==================================================
6. DUPLICATE PREVENTION / UPSERT RULES
==================================================

Use validation and duplicate-safe logic.

--------------------------------------------------
A. ATTENDEE IMPORT
--------------------------------------------------

Primary duplicate rule:
email + eventCode

Behavior:
- if attendee already exists for that event -> update existing record
- if not found -> create new attendee

Do not create duplicate attendee records for the same email/event combination.

--------------------------------------------------
B. SPONSOR IMPORT
--------------------------------------------------

Primary duplicate rules:
- sponsorName for sponsor entity
- sponsorName + eventCode for event-specific sponsorship assignment

Behavior:
- if sponsor exists -> update sponsor master fields
- if sponsor/event assignment exists -> update that event assignment
- if not found -> create new sponsor or sponsor-event assignment

--------------------------------------------------
C. MEETING IMPORT
--------------------------------------------------

Use a reasonable unique matching rule such as:
eventCode + sponsorName + attendeeEmail + date + time + type

Behavior:
- if meeting already exists -> reject or update depending on implementation choice
Preferred:
- reject duplicates unless there is a clear update mode

==================================================
7. VALIDATION RULES
==================================================

Validate before import.

Examples:
- required columns missing
- unknown eventCode
- unknown sponsorName for meeting import
- invalid email format
- invalid date/time format
- invalid meeting type
- invalid status
- invalid sponsorship level
- missing attendee email for attendee/meeting import

If any row fails validation:
- do not silently ignore it
- report the row and the reason

==================================================
8. IMPORT RESULTS / ERROR REPORTING
==================================================

After import, show a clear summary:

Example:
- 84 rows processed
- 71 imported
- 9 updated
- 4 rejected

Also provide a row-level rejection list with:
- row number
- key identifier (email, sponsor name, etc.)
- rejection reason

Examples:
- Unknown eventCode: XYZ2026
- Duplicate attendee for email + event
- Missing required field: email
- Invalid sponsorship level: DiamondPlus
- Unknown sponsor for meeting row

If practical, allow admins to download rejected rows as CSV with an added "errorReason" column.
