Audit Attendee UI Phases A, B, C, and D
Verify what was actually implemented, what is live, what routes are using legacy pages, and what remains incomplete

The attendee screens in production still do not reflect the phased attendee UI work that was specified.

At this point, we need to stop issuing more feature instructions and instead perform a structured implementation audit of the attendee experience.

The goal is to determine:

what parts of Phases A, B, C, and D were actually built

what parts are only partially built

what routes/components are still rendering legacy attendee pages

what is missing from production entirely

what needs to be corrected next in a clean sequence

This audit should be explicit and evidence-based.

11. Verify Event Interest Topics Integration

Since attendee recommendations depend on Event Interest Topics, audit whether the attendee UI is actually using the new topic model.

Required checks

Are attendee interest chips coming from approved active event interest topics?

Or are they still coming from old sponsor-derived text / legacy category source?

Are sponsor recommendations using sponsor_interest_topic_selections?

Are session recommendations using session_interest_topic_selections?

This must be explicitly verified.

12. Verify Recommendation Logic vs Static Rendering

The screenshots suggest the current attendee page may still be mostly static.

Audit whether recommendation blocks are truly recommendation-driven or just rendering available records.

Examples:

Are sponsors ranked by topic overlap?

Are sessions ranked by topic overlap?

Is relevance text shown?

Or are these just static lists?

For every “recommended” element, verify whether logic exists or the label is just cosmetic.

13. Produce a Production Gap Summary

After the audit matrix, provide a short section:

Production Gaps

List the top gaps preventing the attendee experience from matching the intended design.

Example format:

Live attendee route still points to legacy page

Dashboard not connected to post-onboarding flow

My Agenda exists in backend but not visible in attendee navigation

Sponsor directory logic is static, not recommendation-driven

Meeting invitations UI missing from production

This gap summary will drive the next corrective implementation phase.

14. Identify the Correct Next Fix Sequence

After the audit, recommend the next build order based on what is actually missing.

The recommendation should be pragmatic.

Example:

Replace live attendee route

Wire onboarding to dashboard

Surface full agenda + My Agenda nav

Connect sponsor detail and request flows

Add My Meetings and invitation responses

Do not assume all phases need to be rebuilt equally. Base this on the audit findings.

15. No Additional Feature Work Until Audit Is Returned

Do not continue implementing more attendee features before this audit is complete.

The immediate deliverable is the audit report itself, not more new code.

If there are quick route fixes discovered during the audit, note them, but first return the full assessment.

16. Acceptance Criteria

This audit is complete when:

the live attendee route is identified

legacy vs new attendee components are identified

every major Phase A/B/C/D feature is marked as implemented / partial / missing

backend vs UI vs production wiring distinctions are clearly documented

Event Interest Topics integration is verified

recommendation logic is verified as real vs static

a production gap summary is provided

a recommended corrective sequence is provided

17. Success Standard

After this audit, we should have a clear, evidence-based picture of the attendee implementation status and know exactly why the attendee screens still look mostly unchanged in production. The result should remove ambiguity and let us correct the right things in the right order.
