Add message_jobs Architecture + Message Runs UI to Email Center

We need to add a structural communication layer to Concierge so Email Center can track not just individual emails, but the job/run/workflow that caused those emails to be sent.

Right now the system can show individual email sends, but as Concierge grows, Converge needs to understand:

why a group of emails was sent

what workflow triggered them

which event they belonged to

whether the run succeeded overall

how many recipients were targeted

how many succeeded or failed

The solution is to add a parent-layer record:

message_jobs

This should sit above individual email activity records.

1. Primary Goal

Create a new messaging architecture that:

groups related email sends into one parent communication job

allows Email Center to show job-level visibility

improves debugging, auditing, and reporting

supports automations, campaigns, manual sends, and scheduled sends

strengthens Email Center as an event communications control system

2. Core Concept

A single communication action should create:

1 message job

many child email activity records if multiple recipients are involved

Example

A campaign sent to 54 attendees should create:

1 message_job

54 email activity records

A manual sponsor report send to 1 sponsor contact should create:

1 message_job

1 email activity record

A deliverable reminder automation run for 8 sponsor reps should create:

1 message_job

8 email activity records

5. Link Email Activity Records to message_jobs

Add a nullable foreign key to the individual sent email records:

message_job_id

This creates the parent-child relationship.

Required behavior

Every new email sent by the system should attempt to link to a message_job.

If a legacy email record exists without a message job, that is acceptable.

New sends going forward should use the new structure.

6. Do Not Replace Existing Email Activity Records

Important:
Do not replace the existing per-email activity table/log.

Instead:

message_jobs = parent communication run

email activity = individual email sends

This preserves current behavior while adding the missing operational layer.

13. Job Creation Rules
For single-recipient sends

Create one message job and one email activity record.

For batch sends

Create one message job and multiple child email records.

The job status should update based on the child sends.

Example logic

if all succeed → COMPLETED

if some fail → COMPLETED with failures or FAILED depending on your status model

if total failure → FAILED

Keep it simple and transparent.

14. Metrics Rollups on message_jobs

Where possible, roll up the child email metrics into the parent job.

For each job, maintain:

recipient_count

sent_count

failed_count

If provider tracking later supports it, also roll up:

delivered_count

opened_count

clicked_count

Do not block initial implementation on advanced provider metrics if they are not ready yet.

15. Use message_jobs in Campaigns and Automations First

If phased rollout is needed, prioritize creating message jobs for:

Campaign sends

Automation runs

Manual sponsor/attendee sends

System-generated one-off emails

This gives immediate operational value.

16. Preserve Legacy Compatibility

Historical email records may not belong to a message job.

That is acceptable.

Required behavior

old email activity still works

new sends create message jobs

Message Runs only shows records where message_jobs exist

Email Activity continues to show all records

Do not try to perfectly backfill everything unless easy and safe.

17. Add Source Visibility in Email Activity

Once message_jobs exists, Email Activity should use it where possible.

For email activity records linked to a job:

show job name

show source derived from message job

Example:

Source: Campaign – FRC2026 Meeting Invitation

Source: Automation – Deliverable Reminder Run

This improves both views.

20. Acceptance Criteria

This work is complete when:

a new message_jobs table/model exists

new email activity records can link to message_job_id

Message Runs UI exists

Message Runs shows one row per communication job

campaigns create message jobs

automations create message jobs

manual sends can create message jobs

each Message Run has a detail view

child sent emails are visible within the Message Run detail

Email Activity can reference the related message job/source where available

legacy email records continue to work without breaking

21. Success Standard

After this implementation, Email Center should be able to answer not only:

what email was sent

but also:

what communication workflow ran

why it ran

which event it belonged to

how many recipients it targeted

whether the run succeeded overall

That is the standard this change should meet.
