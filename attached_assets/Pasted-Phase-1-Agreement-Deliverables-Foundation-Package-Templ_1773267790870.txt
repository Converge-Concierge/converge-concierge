Phase 1 — Agreement Deliverables Foundation + Package Templates Admin

Build the Phase 1 foundation for the new Agreement Deliverables system in Concierge.

This phase is focused on:

data structure

admin package template management

sponsor agreement generation from templates

negotiated override support

Do not build sponsor-facing deliverables UI yet.
Do not build Cloudflare R2 uploads yet.
Do not build weekly reminder emails yet.

This phase should give us a clean admin-side system that Julie can manage later.

Primary Goal

Create a structured system where:

each sponsorship level has an editable package template

each package template contains structured deliverable items

assigning a sponsor to a package generates a sponsor-specific agreement deliverables set

sponsor-specific negotiated changes can be made without changing the underlying template

This is the core architecture for Agreement Deliverables.

Scope for Phase 1
Build now

Agreement Deliverables admin module shell

Sponsorship Package Templates admin UI

Deliverable Template Items model and editor

Sponsor Agreement Deliverables data model

Generate sponsor agreement deliverables from template

Override / custom deliverable support

Admin list and detail screens

Do not build yet

sponsor dashboard Deliverables tab

sponsor editing/uploading

Cloudflare R2 file upload/storage integration

weekly reminder emails

Zoho Projects sync

advanced reporting

automation beyond initial generation

8. Build Sponsor Agreement Generation Logic

When a sponsor is assigned to a package template for an event, the system should generate the sponsor-specific agreement deliverables.

Create a reusable service function like:

generateAgreementDeliverablesFromTemplate(sponsorId, eventId, packageTemplateId)

This should:

load the selected package template

load all active deliverable template items

create sponsor/event-specific agreement_deliverables

copy template values into the live instance

set is_overridden = false

set is_custom = false

Do not regenerate automatically if agreement deliverables already exist unless explicitly triggered.

16. Do Not Build Files Yet

Even though some deliverables will eventually support files and links, do not build upload/download/file management in Phase 1.

For now:

keep fulfillment_type in the model

keep placeholders for file/link support later

do not integrate R2 yet

This phase is about structure and admin workflow only.

17. Do Not Build Sponsor-Facing UI Yet

Do not add the Agreement Deliverables tab to the Sponsor Dashboard in this phase.

However, structure the data and wording so a sponsor-facing tab can be added cleanly in Phase 2.

18. Keep Zoho Projects Out of Scope

Agreement Deliverables should remain a sponsor-facing and admin-facing communication/fulfillment layer.

Do not try to:

sync Zoho tasks

duplicate Zoho workflows

build task dependencies

build internal project management logic

Julie will continue managing execution in Zoho Projects.

19. Acceptance Criteria

Phase 1 is complete when:

Agreement Deliverables admin module exists

admins can create/edit/archive package templates

package templates can contain structured deliverable template items

admins can assign/generate sponsor agreement deliverables from a template

sponsor agreement detail screens exist

admins can override live agreement deliverables without affecting the template

admins can add custom deliverables

initial Platinum / Gold / Silver / Bronze package structures are supported

20. Success Standard

After Phase 1, Julie should be able to:

maintain standard sponsorship package templates

assign a package to a sponsor for an event

generate the live agreement deliverables

adjust negotiated differences cleanly

update live agreement items without changing the master package

That is the standard this phase should meet.
