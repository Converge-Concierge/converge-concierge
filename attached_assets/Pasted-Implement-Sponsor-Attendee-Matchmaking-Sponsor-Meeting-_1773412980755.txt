Implement Sponsor / Attendee Matchmaking + Sponsor Meeting Request Tool

We want to add a new matchmaking and meeting invitation system to Concierge that helps sponsors identify the most relevant attendees and send controlled meeting requests.

This feature must respect the event experience and attendee privacy.

Sponsors should be able to:

browse recommended attendees

filter attendees by category and relevance

send a limited number of meeting invitations

see invitation status

never receive contact information unless the attendee proactively engages

Attendees do not need a login for this to work.

Instead, attendees will respond through secure email-driven invitation links.

Primary Goal

Create a system where:

attendees are categorized into:

Practitioner

Government / Non-Profit

Solution Provider

sponsors can browse and prioritize attendees in this order:

Practitioners first

Government / Non-Profit second

Solution Providers third

sponsors can send a limited number of meeting invitations based on sponsorship level

attendees receive those invitations through secure email links and can:

accept

decline

choose a meeting time

sponsors do not receive attendee contact information unless:

the attendee schedules a meeting

or the attendee submits an information request

1. Normalize Attendee Category From Eventzilla Ticket Type

We currently receive ticket type data from Eventzilla/Zapier, not a perfectly normalized attendee category.

Examples of ticket types:

Practitioner - Individual Registration

Practitioner - Team Registration (3 or More People)

Solution Provider - Individual or Group Registration

Government - Individual Registration

Required behavior

Add a mapping layer during attendee import / Zapier ingestion.

Create and store:

attendee_category

Supported normalized values:

PRACTITIONER

GOVERNMENT_NONPROFIT

SOLUTION_PROVIDER

Mapping rules

If ticket type contains:

Practitioner → PRACTITIONER

Government → GOVERNMENT_NONPROFIT

Non-Profit → GOVERNMENT_NONPROFIT

Solution Provider → SOLUTION_PROVIDER

Also continue storing the original ticket type:

ticket_type

So Concierge has both:

raw source value

normalized category for matchmaking

2. Create Matchmaking Priority Logic

Attendee category should be the strongest ranking signal for sponsor matchmaking.

Base category weights

Use these default weights:

PRACTITIONER = 100

GOVERNMENT_NONPROFIT = 70

SOLUTION_PROVIDER = 20

This reflects sponsor preference.

3. Add Additional Match Factors

Build a simple scoring engine that combines category priority with other relevant signals.

Recommended scoring inputs:

attendee_category

attendee interests / topic selections

sponsor categories / “what they want to sell”

attendee role/title

optional company type if available

Example match score formula

Use something like:

match_score =
category_weight +
interest_match_score +
role_relevance_score
Recommended starting weights

attendee category = highest weight

topic/category match = medium-high weight

role relevance = medium weight

Keep the first version simple and explainable.

Overview

Default sorting should prioritize attendees by:

category priority

match score

relevance of interests

Filters to include

“Recommended Only”

Recommended matches first

7. Add Sponsor Invitation Limits by Sponsorship Level

Create invitation quotas by sponsorship level.

Recommended defaults:

Platinum = 25

Gold = 15

Silver = 10

Bronze = 5

Store this as configurable sponsorship-level logic if possible.

Required behavior

Sponsors should see:

Invitations Remaining: 12 of 15

Prevent sending invitations once the quota is reached.

This becomes part of the sponsor benefit structure.

9. Add the Sponsor Invitation Flow

When a sponsor clicks Invite to Meeting:

Open a modal with:

attendee name

sponsor name

optional short message

optional topic/reason

current invitation count remaining

Buttons:

Cancel

Send Invitation

When sent:

create a meeting_invitations record

decrement remaining invitation count

send invitation email to the attendee

show success message

Do not create a meeting yet.

10. Build the Attendee Invitation Email Flow (No Login Required)

Attendees do not have logins, so invitations must work through secure email links.

Email subject example
A Sponsor Would Like to Meet With You at [Event Name]
Email body should include

sponsor name

sponsor description or topic focus

optional invitation message

Accept Meeting button

Decline button

Each button should use a secure signed token link.

11. Create Secure Attendee Invitation Pages

Example route patterns:

/invite/:token

/meeting-invitation/:token

Token should resolve:

attendee

sponsor

event

invitation record

expiration

Do not require account creation or login.

13. Build Attendee Meeting Scheduling After Acceptance

If attendee clicks Accept Meeting:

Show a simple scheduling page where they can:

pick from available meeting slots

select onsite or online if relevant

confirm the meeting

Then:

create the actual meeting record

mark invitation as scheduled/accepted

notify sponsor

optionally send confirmation email

This should reuse existing Concierge meeting scheduling logic where possible.

14. Protect Attendee Contact Information

This is critical.

Sponsors must not receive attendee contact information unless the attendee proactively engages.

Rule

Sponsors only get attendee contact details if:

attendee schedules a meeting

or attendee submits an information request

Before that, the sponsor may only see:

name

title

company

interests/category

relevance context

Do not expose email or phone.

15. Add Invitation Limits Per Attendee (Optional but Recommended)

To protect the attendee experience, prevent a practitioner from receiving too many invitations.

Recommended starting rule:

max 6 sponsor invitations per attendee per event

If limit reached:

additional invitation attempts are blocked

or the attendee is deprioritized in sponsor discovery

This keeps the experience curated.

In event settings, add matchmaking controls such as:

20. Acceptance Criteria

This feature is complete when:

attendee category is normalized from Eventzilla ticket type

sponsors can browse attendees in Attendee Discovery

attendees are ranked with Practitioners first

sponsors can send meeting invitations

invitation count is limited by sponsorship level

attendees receive secure email invitations without needing a login

attendees can accept/decline and schedule meetings

sponsors do not receive contact info unless attendee proactively engages

invitation analytics are visible

admin can monitor invitation activity

21. Success Standard

After implementation, Concierge should function as a curated sponsor-to-attendee matchmaking platform, where sponsors can proactively invite the most relevant attendees to meet, while preserving attendee control and privacy.

That is the standard this feature should meet.
